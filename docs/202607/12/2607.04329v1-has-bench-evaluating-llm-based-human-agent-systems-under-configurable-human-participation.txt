## ABSTRACT 

Large language models increasingly operate in settings where humans are ac-tive collaborators rather than passive task providers. We introduce HAS-FRAMEWORK , a graph-based framework that represents humans and LLM-powered agents as first-class participants with explicit roles, permissions, commu-nication paths, and action authority. Building on this framework, HAS-B ENCH 

evaluates Human-Agent Systems under configurable human participation across agency levels, interaction channels, and persona policies. The benchmark mea-sures both task outcomes and process-level collaboration behavior, including clar-ification quality, feedback utilization, control calibration, safety, initiative, and in-teraction cost. Experiments across six domains show that human participation can substantially improve task completion and failure recovery, but the gains depend on when, how, and by whom human input is exercised. 

## 1 INTRODUCTION 

Recent advances in Large Language Models (LLMs) have enabled a new generation of Human-Agent Systems (HAS), which integrate the strengths of both humans and LLM-based agents to collaboratively solve complex tasks (Zou et al., 2025). Unlike conventional agentic systems that often treat users as passive information providers (Wang et al., 2024a; Durante et al., 2024), HAS support dynamic, bidirectional collaboration, where the human can shape the agent’s actions and decisions through clarification , feedback , and control , exercised both at the agent’s request and on the human’s own initiative (Shao et al., 2024; Miao et al., 2025; Barres et al., 2025). Such systems are increasingly explored in real-world scenarios, where human expertise, contextual judgment, or real-time intervention is critical, including travel planning (Qian et al., 2025), financial decision-making (Xu et al., 2025), and autonomous driving (Wu et al., 2025). Despite this growing interest, existing benchmarks only partially capture the full interaction dynam-ics of HAS. General agent benchmarks (Liu et al., 2024; Zhou et al., 2024a; Xie et al., 2024; Trivedi et al., 2024) evaluate autonomous agents in realistic interactive environments, but typically treat users as static task providers rather than active collaborators. User-agent interaction bench-marks (Wang et al., 2024b; Lu et al., 2025; Qian et al., 2025; Barres et al., 2025) introduce multi-turn dialogue, tool use, preference elicitation, or dual-control settings, yet they largely focus on single agent-user scenarios with fixed user roles, and do not systematically examine how varying of human participation affect coordination, safety, and task completion. In contrast, multi-agent benchmarks (Zhu et al., 2025; Zhou et al., 2024b; Sun et al., 2025; Emde et al., 2026) study co-ordination and competition among LLM agents, but rarely model humans as first-class participants with explicit agency, permissions, responsibilities, and intervention channels. Thus, an agent must decide not only what action to take, but also when to ask for input and how to incorporate it from different human collaborators. 

## 2 RELATED WORK 

Benchmarks for Autonomous LLM-based Agents. LLM agent evaluation has shifted from static language understanding to interactive task execution, where agents reason, plan, invoke tools, and modify external states over multiple steps. Benchmarks now span diverse environments and applica-tions (Liu et al., 2024; Zhou et al., 2024a; Trivedi et al., 2024), multimodal computer use (Xie et al., 2024), long-horizon assistant tasks (Mialon et al., 2023), software engineering (Jimenez et al., 2024), and tool-augmented reasoning and API invocation (Qin et al., 2024; Li et al., 2023). These bench-marks establish important foundations for evaluating LLM agents as autonomous decision makers in interactive environments. However, their primary focus remains on whether an individual agent can complete a task given an instruction, tool interface, or environmental feedback. The human user is typically modeled as a static task provider, an implicit source of preferences, or an external evaluator, rather than as an active collaborator who can shape task execution through information, feedback, or intervention. 

Benchmarks for LLM-Based Multi-Agent Systems. Beyond single-agent settings, LLM-based multi-agent systems introduce additional challenges of communication, specialization, and coor-3dination, as multiple agents jointly solve tasks or simulate social interactions. SOTOPIA (Zhou et al., 2024b) evaluates social intelligence through goal-driven agent interactions, while MultiA-gentBench (Zhu et al., 2025) supports multiple coordination protocols and process-aware metrics for planning, communication, and individual contribution. Collab-Overcooked (Sun et al., 2025) fur-ther studies collaboration in shared task environments, emphasizing process-level properties such as proactive coordination, execution efficiency, and adaptation over time. MASEval (Emde et al., 2026) shifts the unit of evaluation from isolated language models to complete multi-agent systems, en-abling comparisons across models, orchestration frameworks, and task environments. These bench-marks primarily target agent-agent interaction: the main participants are LLM agents with assigned roles and communication protocols, while humans are typically outside the coordination loop. They therefore leave open how to evaluate coordination when humans actively provide context, revise goals, or intervene during execution, motivating benchmarks that bring users into the evaluation loop. 

Benchmarks for User-Agent Interaction. A growing line of benchmarks explicitly models user-agent interaction. MINT (Wang et al., 2024b) evaluates multi-turn tool use with natural-language feedback from simulated users, and ToolSandbox (Lu et al., 2025) adds stateful conversational tool use in a persistent environment, where earlier actions can affect later tool outcomes. User-Bench (Qian et al., 2025) models user goals as underspecified, incremental, and indirect, requiring agents to clarify intent and elicit preferences. PeopleJoin (Jhamtani et al., 2025) moves beyond a single user, requiring an agent to coordinate with multiple human collaborators and gather informa-tion distributed across them. For service tasks, τ -bench (Yao et al., 2024) evaluates tool-agent-user interaction under domain policies, and τ 2-Bench (Barres et al., 2025) extends it to a dual-control environment where both agent and user act on a shared state. Collaborative Gym (Shao et al., 2024) studies human-agent collaboration in shared task environments, supporting bidirectional and non-turn-taking interaction as well as process-level evaluation. Although these benchmarks represent important progress toward realistic user-agent evaluation, most prior work still focus on single user-agent dyads, predefined user roles, or selected forms of human involvement, rather than broader HAS in which humans and multiple agents jointly co-ordinate under different roles, communication structures, and intervention policies. HAS-B ENCH 

addresses these limitations by making human participation a configurable evaluation setting (Shao et al., 2025; Zou et al., 2025). It represents humans and agents as participants in the same inter-action graph, factorizes human involvement into clarification , feedback , and control channels, and varies agency levels and personas across diverse coordination structures and shared environments (as summarized in Table 1). This design enables controlled comparison of how different forms and degrees of human participation affect not only task success, but also collaboration quality, safety, intervention effectiveness, and interaction cost. 

## 3 HAS-F RAMEWORK 

Existing agent and multi-agent frameworks often support human involvement through external su-pervision mechanisms, such as runtime interrupts, approval gates, or reviewer-style checkpoints (Wu et al., 2024; Takerngsaksiri et al., 2025). However, these mechanisms typically keep humans outside the system, treating human participation as a late-stage supervision signal rather than a designable and schedulable component of the system. This treatment of humans as external supervisors is in-sufficient for HAS, where humans may not only approve or correct agent behavior, but also actively shape task execution in ways agents cannot fully replace (Zou et al., 2025). To address this limita-tion, we introduce HAS-Framework , a graph-based framework centered on the HAS Graph , where humans are represented as nodes in the same interaction structure as LLM-powered agents, with ex-plicit roles, responsibilities, and permissions. HAS-B ENCH builds on this framework to instantiate controlled scenarios for outcome-level and process-level evaluation. 3.1 HAS G RAPH : H UMANS AND AGENTS AS UNIFIED PARTICIPANTS 

HAS-Framework instantiates a typed directed graph G = ( V, E), where the node set V = VA ∪ V H

consists of LLM-powered agent nodes VA and human participant nodes VH . Unlike agent-only graphs that define coordination solely among LLM agents (Zhu et al., 2025), the HAS Graph places humans and agents in the same interaction structure. Human and agent nodes share a common rep-4resentation with role-specific attributes, such as responsibilities, capabilities, tool permissions, and visibility scopes, which determine what each participant can observe, decide, and execute. Each edge in E is a typed interaction relation that specifies how participants may communicate, delegate, supervise, hand off tasks, or intervene during execution. This graph-native representation removes the asymmetry between system-internal agents and system-external users, making human participa-tion part of the scenario configuration rather than an external supervision mechanism. By encoding human participation at the graph level, the HAS Graph makes it both designable and 

schedulable . At design time, a scenario specifies where humans are needed, what roles they play, and what responsibilities they hold. At runtime, typed edges route requests to the appropriate human participant rather than to a generic “human input” interface. As a result, humans are not limited to late-stage approval or error correction, but can act as domain experts, safety reviewers, task owners, collaborators, or user-side tool operators who contribute specialized knowledge, contextual judg-ment, authorization decisions, or actions that agents may not be able or allowed to perform. 3.2 HUMAN INTERACTION CHANNELS : C LARIFICATION , F EEDBACK , AND CONTROL 

A human participant in HAS may provide missing information, monitor or repair intermediate re-sults, or retain authority over decisions and actions. Treating various human behaviors as a single generic “human input” would obscure their distinct roles in collaboration. Prior work in human factors suggests that human involvement in automated systems spans multiple functions, includ-ing information acquisition, information analysis, decision/action selection, and action implemen-tation (Parasuraman et al., 2000). Similarly, studies of joint activity emphasize that effective col-laboration requires participants to establish shared understanding, monitor and repair intermediate progress, and remain directable by one another (Klien et al., 2005; Klein et al., 2005). Motivated by these perspectives, HAS-F RAMEWORK operationalizes human involvement into three controllable interaction channels: clarification , feedback , and control . Detailed definitions are provided in Ap-pendix A.1. Clarification targets input-level uncertainty. It is used to resolve missing, ambiguous, conflicting, or underspecified information before the system commits to a plan, tool call, or final re-sponse. Clarification helps humans and agents establish sufficient shared context for task execution. 

Feedback targets output-level quality. It allows humans to evaluate, correct, or refine intermediate plans, decisions, generated artifacts, or partial results. Feedback is especially useful when the task requires iterative improvement, subjective judgment, or verification against criteria that are difficult to fully specify upfront. Control governs action-level execution. It enables humans to approve, veto, modify, override, take over, or directly execute actions that affect the shared environment. Control is important not only for protected or safety-sensitive operations, but also for actions that require human-only permissions, credentials, capabilities, physical access, or user-side tools. In short, clarification acts on input , feedback acts on output , and control acts on action . Each channel can be agent-initiated or human-initiated. By making these channels explicit, HAS-F RAMEWORK 

turns human involvement into configurable and traceable events, enabling HAS-B ENCH to ablate and evaluate different forms of human participation. 3.3 AGENCY LEVELS : C ONFIGURING HUMAN AUTHORITY 

HAS-F RAMEWORK configures human authority along the five-level Human Agency Scale (A1– A5) (Shao et al., 2025; Zou et al., 2025), ranging from full automation, where the agent acts alone, to human-driven workflows, where the human stays continuously in control. Levels A1– A2 are automation-oriented , with the agent replacing most human effort, whereas A3–A5 are 

augmentation-oriented , with the human retaining meaningful involvement (as illustrated in Table 2). HAS-F RAMEWORK maps each agency level to a human-agency policy that specifies agent prompt, available C/F/C channels, triggering conditions, and human authority over decisions and actions. This turns an abstract agency level into a concrete, reproducible runtime configuration, letting HAS-Bench instantiate the same task under different levels and isolate the effect of human participation on outcomes and collaboration dynamics. In HAS-B ENCH , we adapt verified tasks from existing benchmarks and augment them with interac-tion requirements that commonly arise in real-world HAS. Drawing on prior human-agent collabo-ration literatures (Qian et al., 2025; Ruan et al., 2024; Zou et al., 2026), we summarize six represen-tative problem patterns : Information Asymmetry , Latent Constraint , Safety-Critical Authorization ,

Multi-Stakeholder Specification , Iterative Verification , and Dynamic Goal Revision . Details of these patterns are provided in Appendix B.1. Each HAS scenario is constructed through a three-step pipeline: 

Step 1: Base task construction. We first select candidate tasks from existing user-agent and multi-agent benchmarks, including τ 2-Bench (Barres et al., 2025) and MultiAgentBench (Zhu et al., 2025), that provide executable environments and verifiable outcomes. To ensure that the resulting 6scenarios are meaningful for HAS, we first apply a rule-based filtering procedure to remove near-duplicate scenarios by grouping tasks along domain-specific distinguishing attributes, such as fault families and fault-count tiers in Telecom, or topic families and team sizes in Research, and retain a bounded number of representative tasks from each group. Within each group, we rank tasks using interaction-richness signals derived from source metadata and task characteristics, including the number and diversity of required tool actions, the diversity of collaborating roles, and the availability of verifiable evaluation assertions. We additionally discard tasks that lack reliable evaluation signals. For each selected task, we preserve the original task goal, initial state, domain tools, environment update rules, and evaluation assertions whenever available. This allows HAS-B ENCH to inherit the realism and verifiability of existing benchmarks while focusing its modifications on human-agent interaction. Overall, this stage reduces 2,749 raw source tasks to 301 representative base tasks. Detailed filtering rules and per-domain statistics are provided in Appendix B.2. 

Step 2: HAS scenario adaptation. For each base task, we identify which problem patterns the task can naturally support, based on its goal, available tools, state structure, and evaluation criteria. This assignment is proposed through four independent LLM judges and then filtered using a conser-vative agreement criterion. The judges evaluate source-grounded adaptation fit, evaluator feasibility, and the risk of introducing unsupported roles, constraints, or objectives, and provide supporting evidence from the source task. We then adapt the base task to instantiate the corresponding pat-tern. Concretely, we instantiate human participants with explicit roles and responsibilities, assign role-specific tool permissions, and define user-side information states, including what each human initially knows, what is hidden from agents, and what can only be revealed through interaction or user-side actions. These adaptations make human participation meaningful and measurable. The complete panel prompt is provided in Appendix B.3.1. 

Step 3: Benchmark Validation and Human Verification. Finally, we validate each adapted sce-nario through a multi-stage quality-control process. We first perform automatic consistency checks to ensure that the scenario is complete, the environment can be initialized, the required tools and permissions are available. We then use a panel of LLM reviewers to independently assess whether each scenario is solvable, realistic, and aligned with the intended interaction requirement. Reviewer judgments are aggregated through majority voting, and scenarios with low agreement or failed cri-teria are revised or removed. To further ensure quality, we conduct sampled human verification across domains and interaction types, checking task realism, interaction naturalness, and evaluation correctness. This validation process ensures that the final HAS-Bench scenarios are executable, meaningful for human-agent interaction, and reliably evaluable. The complete validation protocol, reviewer prompts, human rubric, and validation statistics are provided in Appendix B.4. HAS-B ENCH evaluates HAS along two complementary dimensions: outcome-level task perfor-mance and process-level collaboration behavior . Outcome-level metrics measure whether the task objective is achieved, while process-level metrics characterize how humans and agents com-7municate, coordinate, and intervene. Detailed metric definitions, judge rubrics and prompts, and aggregation procedures are provided in Appendix B.5. 

Outcome-level metrics. We measure task performance with pass@ k (Chen et al., 2021), the un-biased estimate of the probability that at least one of k sampled executions succeeds, where a run succeeds only if all task dimensions pass. We report pass@1 under deterministic decoding as the single-run success rate. To capture partial completion and non-delivery failures, we additionally report Task Score (TS) , the average partial credit over the task-specific dimensions, and Delivery Rate (DR) (Shao et al., 2024), the fraction of episodes in which the system produces task-relevant output within the step budget. In addition, Safety Rate measures outcome-level safety, i.e., whether the final state satisfies domain-specific safety assertions, such as avoiding unauthorized, harmful, or policy-violating outcomes. Finally, HAS Rescue Rate (HAS RR) reports the fraction of tasks that, having failed under autonomy, are recovered once a human participates. 

Process-level collaboration metrics. Process-level metrics measure how human participation contributes to task execution. The Clarification Quality Score (CQS) rates whether agent-initiated clarifications are precise, necessary, and non-redundant. The Feedback Utilization Rate (FUR) 

rates whether human feedback was actionable and actually integrated by the agent in subsequent planning, editing, or execution. The Control Request Justification (CRJ) rates whether re-quests for human authorization are well-timed and warranted for high-stakes, ambiguous, or policy-sensitive actions, rather than over-asking on trivial ones. The Action Safety Rate (ASR) instead audits the process : it checks every agent action against a set of unsafe behaviors and marks a task safe only if all its actions are safe, catching unsafe behavior during execution even when the final state is safe. Finally, we use two metrics to characterize the overall collaboration: Initiative En-tropy (Hinit ) (Shao et al., 2024) measures how evenly initiative is distributed across participants, and the Human Intervention Rate (HIR) Feng et al. (2024) measures the share of human steps in the interaction, reflecting the overall intensity of human involvement. We additionally track interac-tion cost through the number of turns, interaction steps, human steps, and tokens. 

## 5 EXPERIMENTS 

We evaluate LLM-based Human-Agent Systems on HAS-Bench to answer four questions: (RQ1) 

How does human participation improve task success, outcome quality, safety, and failure recovery over autonomous execution? (RQ2) How do different levels of human agency affect outcome-level performance, process-level collaboration quality, and human intervention cost? (RQ3) How do forms of human-agent interaction vary across different problem patterns? (RQ4) How do different user personas influence task outcomes, collaboration dynamics and interaction cost? 5.1 EXPERIMENTAL SETUP 

Models, user simulator, and evaluation judges. We use GPT-4.1 as the default LLM-judge backbone. We focus the main experiments on three representative operating points along the automation-to-augmentation transition: full automation (A1), equal partnership (A3), and agent-assisted (A4). We contrast A3 against A1 to quantify the value of human participation and include A4 to examine whether proactive human initiative provides additional benefits beyond reactive equal partnership. We omit A2 (Minimal Human Input) and A5 (Human-Driven): A2 is a narrow automation-regime variant adjacent to A1, while A5 reduces the agent to an assistant under continuous human control, so neither adds much signal about human-agent collaboration beyond the A1/A3/A4 span. 

Evaluation protocol. We follow the evaluation protocol of Section 4.3. As outcome metrics we report pass@1 (task success), Task Score , Delivery Rate , and Safety Rate . For A1–A3 compar-isons, we additionally report the HAS Rescue Rate : the fraction of tasks that fail under A1 but succeed under A3. Process-level metrics, computed from execution traces, characterize how col-laboration unfolds: Clarification Quality Score (CQS), Feedback Utilization Rate (FUR), Control Request Justification (CRJ), Action Safety Rate (ASR), Initiative Entropy ( Hinit ), and Human Inter-vention Rate (HIR). CQS, FUR, and CRJ are event-conditioned: each score is computed only over traces containing the corresponding clarification, feedback, or control-request event. To separate channel usage from event quality, we report each conditional score together with event coverage and the mean number of events per task. Traces without the corresponding event are treated as undefined. We further track interaction cost in interaction turns, tool calls, human steps, and tokens. 5.2 MAIN RESULTS

5.3.1 INTERACTION CHARACTERISTICS ACROSS PROBLEM PATTERNS (RQ3) The agency sweep shows that stronger human agency changes both outcomes and collaboration dy-namics. We next examine the form of interaction at a finer granularity. HAS-Bench defines six problem patterns during scenario construction, each corresponding to a different interaction require-ment in human-agent collaboration. We use the three C/F/C channels as diagnostic probes. In this diagnostic ablation, we keep the backbone model (GPT-4.1), decoding setting, task set, and evalua-tion protocol fixed, and vary only which human-interaction channels are available. Because different problem patterns have different diagnostic objectives, we evaluate them with natural metrics. For Information Asymmetry, Latent Constraint, Multi-Stakeholder Specification, Iterative Verification, and Dynamic Goal Revision, we report Recovery Rate : the fraction of A1-failed tasks in the pat-tern that are solved under a given channel condition. For Safety-Critical Authorization, we instead report Safety Rate : the fraction of valid safety-critical tasks in which no protected action is exe-cuted without human approval. These denominators are therefore pattern-conditioned rather than identical across rows; the table should be read as a within-pattern diagnostic comparison, not as a cross-pattern prevalence estimate. Table 5 shows clear differences in the dominant single-channel condition across problem patterns. For Information Asymmetry and Latent Constraint, the strongest channel is clarification: these pat-terns require the agent to surface missing facts, hidden requirements, or acceptance criteria before committing to an action or output. Feedback is strongest for Multi-Stakeholder Specification, It-erative Verification, and Dynamic Goal Revision, where human input often repairs intermediate outputs, reconciles competing requirements, validates partial progress, or adapts the task after ex-ecution has begun. In Safety-Critical Authorization, Cont-only reaches a Safety Rate of 100% , far above Clar-only and Feed-only, showing that clarification or feedback cannot substitute for explicit human authorization over protected actions. In addition, clarification and feedback are related but not interchangeable. In Information Asymme-try, Clar-only and Feed-only obtain similar recovery rates ( 48% vs. 44% ), suggesting that feedback can sometimes compensate for missing information after an incorrect or incomplete attempt. In La-tent Constraint, however, Clar-only substantially outperforms Feed-only ( 29% vs. 17% ), indicating that hidden acceptance criteria are more useful when elicited before the agent commits to a candi-date solution. In this experiment, we hold the task set, model, agency level, permissions, private information, and evaluation criteria fixed, and vary only the user interaction policy. We use the same GPT-4.1/A3 evaluation setting as the main collaboration experiments unless otherwise specified. Each domain defines three domain-specific personas, summarized in Appendix C.5 (Table 17). Table 6 shows that user personas can substantially change outcomes even when the task and agency setting are fixed. The largest outcome sensitivity appears in domains where the user interaction pol-icy can strongly shape the task trajectory. Bargaining has the largest Pass@1 range ( 63 .0 points), reflecting that negotiation behavior can determine whether the interaction reaches a successful agree-ment. Coding and Research also show large sensitivity: Coding has a 23 .0-point Task Score range and a 21 .8-point Pass@1 range, while Research has a 21 .7-point Task Score range and a 33 .4-point Pass@1 range. By contrast, the structured service domains show smaller outcome ranges, with Pass@1 ranges below 7 points and Task Score ranges below 8 points. The process metrics reveal a different aspect of persona sensitivity. HIR ranges are small across all domains, at most 0.04 , indicating that personas do not primarily change the overall fraction of human steps. Instead, personas change how the interaction unfolds. Bargaining has the largest range in both turns ( 4.4) and agent asks ( 3.77 ), while Telecom shows a large ask range ( 1.97 ) despite small outcome variation. Coding also shows a substantial ask range ( 1.52 ), suggesting that different user archetypes can lead the same agent to ask substantially more or fewer clarification questions. Thus, persona effects may be hidden if we only measure final task success: different personas can induce different asking behavior and interaction costs even when final outcomes are relatively stable. Together with the pattern-level channel analysis in Section 5.3.1, these results show that Human-Agent Systems are sensitive along both task-side and human-side axes. Problem patterns determine which forms of human participation are most useful, while personas affect how costly, direct, or robust those interactions are in practice. This suggests that a fixed participation policy is insufficient for robust Human-Agent Systems: agents must decide not only what action to take, but also when to ask and how to incorporate input from different human collaborators. 13 6 CONCLUSION 

We introduced HAS-F RAMEWORK and HAS-B ENCH for evaluating LLM-based Human-Agent Systems under configurable human participation. HAS-F RAMEWORK represents humans and LLM agents as first-class participants in a shared interaction graph, while HAS-B ENCH instantiates this framework into 397 tasks across six domains with configurable agency levels, interaction chan-nels, and user personas. Unlike benchmarks that primarily measure autonomous task completion, HAS-B ENCH evaluates both final outcomes and process-level collaboration dynamics, including clarification, feedback, control, safety, initiative, intervention, and interaction cost. Our experiments show that structured human participation can improve task success, outcome qual-ity, safety, and failure recovery, but its benefits are conditional. Capable agents benefit more from equal partnership, while weaker agents may fail to effectively use human input. More proactive hu-man agency can further improve outcomes, but may also introduce regressions when interventions are premature or distracting. Channel ablations reveal that different problem patterns require differ-ent forms of participation, and persona analysis shows that human interaction styles can substantially affect outcomes and cost. These findings suggest that effective Human-Agent Systems require more than adding a human into the loop. Agents must learn when to ask for input, when to seek feedback or authorization, how to incorporate human contributions, and how to adapt to different collaborators.

A.1 OPERATIONAL DETAILS OF HUMAN INTERACTION CHANNELS 

The human interaction channels are distinguished by the primary object on which human partic-ipation acts: clarification acts on information needed for execution, feedback acts on an existing intermediate output, and control acts on action execution. These channels are operationally separa-ble but may co-occur within the same episode. 

Layered implementation. In HAS-B ENCH , clarification, feedback, and control share the same three-layer implementation structure. The protocol layer defines typed data objects; the user-behavior layer defines how a simulated human responds under a persona, and the orchestration layer routes, gates, records, and resolves interaction events. This separation keeps channel seman-tics independent from domain tools and makes channel-level ablations possible without changing the underlying task environment. At the protocol layer, each channel is represented by typed request and response objects. For exam-ple, clarification uses a request object for agent-to-user questions and a response object containing the user’s answer. Feedback uses response objects that may contain comments, ratings, labels, ed-its, or demonstration actions. Control uses request and decision objects, where the decision may approve, deny, stop, override, take over, or route execution to a human participant. At the user-behavior layer, each channel has an abstract responder interface and an LLM-backed implementation that constructs a persona-aware prompt. The responder does not directly call the LLM, it produces the channel-specific prompt. Persona dimensions are selectively used by channel: clarification behavior is primarily shaped by patience and cooperativeness; feedback additionally uses assertiveness; and control uses risk tolerance, assertiveness, and cooperativeness. At the orchestration layer, each channel controller tracks request lifecycle state. A request is first created and validated, assigned a unique identifier, stored as pending, dispatched to the appropriate human participant through the HAS Graph, and then resolved as answered, expired, or otherwise terminated. 

Availability and initiation. A channel is available when it is enabled by the current human-participation setting and permitted by the corresponding edge in the HAS Graph. Availability does not imply that the channel must occur in every episode; actual use depends on the task trajectory. Each channel may be initiated in two ways: • Agent-requested. The agent initiates an interaction by asking a human participant to provide missing information, review an intermediate output, authorize or modify a proposed action, or directly perform an action that the agent cannot or is not permitted to execute. • Human-initiated. The human initiates an interaction without a corresponding pending agent request by proactively supplying or correcting information, evaluating or revising an intermediate output, or approving, vetoing, overriding, taking over, or directly executing an action. 

Virtual channel tools and execution flow. C/F/C interactions are exposed to the agent as virtual channel tools. Unlike domain tools, which modify an external environment such as a database, workspace, or service state, virtual channel tools create typed interaction events and then pause agent execution. The orchestrator resolves the event by routing it to the appropriate human participant, collecting the response or decision, recording it in the trace, and writing the result back to the agent memory before execution resumes. The end-to-end control flow is the same at a high level across the three channels. When the agent calls a virtual channel tool, the agent runtime converts the tool call into a pending channel event and pauses rather than returning a domain-tool result. The orchestrator then dispatches the event to the corresponding channel handler. The handler builds a typed request through the channel controller, checks the current human-participation policy and interaction budget, routes the request to connected human nodes in the HAS Graph, formats it as a user-facing message, and invokes the shared user-simulator path. The user response is then finalized into a structured response or decision, written back to the agent memory, and appended to the execution trace. 18 Table 7: Operational semantics of the three human interaction channels. Event classification and disambiguation. Because the same utterance may superficially resemble more than one channel, we classify each interaction event according to its primary operational effect on the task trajectory, rather than its surface form. 1. If the event determines whether, how, or by whom a proposed, pending, or ongoing task action is executed, it is classified as control . This includes authorizing, blocking, modifying, replacing, redirecting, transferring, halting, taking over, or directly performing an action, including actions that the agent is unable or not permitted to execute. 2. For example, “Do not send that message” is a control veto when it blocks a pending send action, whereas “The draft is too informal, please revise it” is feedback because it evaluates an existing artifact. Similarly, asking a human to log in, provide a signature, or execute a user-side tool is control because it transfers action execution to the human; asking which account, document, or option should be used is clarification because it resolves missing information. A corrected factual input is classified as clarification when it updates the information state for subsequent reasoning, whereas a critique of an output produced from that information is classified as feedback. ## B HAS-B ENCH 

B.1 PROBLEM PATTERN DETAILS 

HAS-B ENCH organizes recurring interaction requirements in Human-Agent Systems into six prob-lem patterns . We derive these patterns from prior work that studies when and why human partici-pation is needed during agent execution. We draw on representative benchmarks and studies across related lines of research: user-agent interaction and preference elicitation, human-agent collabora-tion, agent safety and authorization, multi-user coordination, iterative feedback and verification, and goal-shifting or interruptible agents. For each line, we ask a single question: what does the task require from the human that the agent cannot supply on its own? 

We keep a requirement as a distinct pattern only if it satisfies three criteria. First, it recurs across multiple independent prior works rather than appearing only in a single benchmark. Second, it corresponds to a distinct interaction requirement, differing in what the human supplies and when it is needed relative to the agent’s actions. Third, it has observable consequences in an executable task setting, so that success or failure can be evaluated rather than only described qualitatively. Crucially, each pattern is defined by the underlying reason human participation is needed, rather than by a particular domain, tool interface, or task format. Table 10 summarizes the six patterns and related prior work. 

Information Asymmetry. This pattern captures cases where task-relevant factual information is available to the human side but not directly observable by the agent. It is motivated by user-centric and ambiguity-aware agent benchmarks, where agents must ask questions or interact with users to recover missing information (Qian et al., 2025; Wang et al., 2025; Vijayvargiya et al., 2026; Kobal-czyk et al., 2025). The key property is that the missing content is factual or state-like: for example, account details, local context, credentials, user-side observations, or environment information that the agent cannot infer reliably from the initial instruction. 21 Table 10: The six problem patterns in HAS-Bench.

Latent Constraint. This pattern captures requirements, preferences, or acceptance criteria that already exist before execution but are not explicitly stated in the initial instruction. It is motivated by work on latent user preferences, user edits, and underspecified software tasks (Gao et al., 2024; Vijayvargiya et al., 2026). Compared with Information Asymmetry, the missing content is less about factual state and more about what would make a solution acceptable, such as style, budget, compatibility, implementation constraints, or hidden evaluation criteria. 

Safety-Critical Authorization. This pattern captures actions that should not be executed solely at the agent’s discretion because they are sensitive, costly, irreversible, harmful, or policy-sensitive. It is motivated by agent-safety and sandboxed-risk benchmarks that study unsafe tool use, harmful computer actions, and the safety–pragmatism trade-off in autonomous agents (Ruan et al., 2024; Zhang et al., 2024; Kuntz et al., 2026; Simhi et al., 2025). The defining feature is that human participation is not merely informative: the human retains authority through approval, veto, override, or direct execution. 

Multi-Stakeholder Specification. This pattern captures tasks where relevant knowledge, prefer-ences, or decision authority is distributed across multiple human participants. It is motivated by multi-user information gathering and collaborative personalization settings, where an agent must coordinate with several people rather than a single monolithic user (Jhamtani et al., 2025; Lee et al., 2025). The central challenge is not only to elicit information, but also to reconcile, prioritize, or route inputs across distinct human roles. 

Iterative Verification. This pattern captures tasks where the goal is relatively stable, but interme-diate or final outputs require human review, testing, validation, or correction over multiple rounds. It is motivated by conversational code generation and collaborative work environments in which feedback is an integral part of task completion (Han et al., 2025; Shao et al., 2024). The defin-ing property is that human participation evaluates and repairs intermediate artifacts rather than only supplying missing initial information. 22 Dynamic Goal Revision. This pattern captures tasks where the user’s goal changes during exe-cution through addition, revision, cancellation, or re-prioritization. It is motivated by interruptible-agent and goal-shift benchmarks, where agents must adapt when users change their mind or when missions evolve over time (Zou et al., 2026; Rana et al., 2025; Yu et al., 2025). The central challenge is maintaining responsiveness to updated human intent instead of rigidly optimizing the original in-struction. 

Non-exclusivity. The six patterns are intended as primary interaction labels rather than mutually exclusive categories. A realistic Human-Agent System may involve several patterns at once: for example, a service task may combine hidden account information with a protected action, or a coding task may combine latent constraints with iterative verification. We use the primary pattern to identify the dominant reason human participation is needed, while allowing secondary interaction properties to appear in scenario metadata and qualitative analysis. B.2 SOURCE TASK FILTERING 

Sources and output. We draw candidate tasks from six domains across two benchmarks that pro-vide executable environments and verifiable outcomes: τ 2-Bench (Barres et al., 2025) for Airline, Retail, and Telecom, and MultiAgentBench (Zhu et al., 2025) for Coding, Research, and Bargaining. For each retained task, we preserve the source task goal, initial state, available tools, environment-transition rules, and evaluation criteria provided by the source benchmark. This preserves the exe-cutable semantics of the original task before the human-agent interaction layer is added. 

Filtering procedure. For each domain, we apply the following three deterministic steps: 1. Domain-specific bucketing. We group tasks according to attributes that distinguish scenario structure within the domain, so that near-duplicate tasks fall into the same bucket. Examples include fault family and fault-count tier in Telecom, write-action category and multi-order status in Retail, and topic family and team size in Research. 2. Bounded retention with richness ranking. We retain a bounded number of representative tasks from each bucket. Tasks are ranked using interaction-richness signals derived from source meta-data, including the number and diversity of required tool actions, the presence of hidden in-formation or constraints, the diversity of collaborating roles, and the availability of checkable evaluation criteria. 3. Eligibility filtering We remove tasks that lack any reliable evaluation signal or fail domain-specific eligibility requirements. Tasks without action-level criteria may still be retained when they contain usable natural-language assertions, in which case they are flagged for downstream validation. B.4 BENCHMARK VALIDATION AND HUMAN VERIFICATION 

Each adapted scenario passes four quality-control components: (i) an automatic check, (ii) LLM-based structural validation, (iii) an LLM-based worthiness review, and (iv) sampled human verifi-cation. The two LLM panels serve complementary purposes: the structural panel tests whether a scenario is internally valid and executable, whereas the worthiness panel tests whether it is realistic, non-trivial, meaningfully dependent on human participation, and experimentally useful. The judgments are aggregated by majority vote for each dimension. A scenario that fails a hard preflight or structural check is returned for repair. A scenario receiving a conditional worthiness verdict is revised and re-evaluated, while an excluded scenario is removed. B.4.1 AUTOMATIC CHECKS 

Each candidate scenario is loaded through the same runtime used in the experiments. The proce-dure: (i) parses the scenario file, (ii) constructs the run configuration, (iii) instantiates the typed configuration object through the framework loader, and (iv) executes the configuration’s validation routine. The checks verify that all required fields are present and correctly typed, the environment 28 can be initialized, all participant identifiers resolve, referenced tools exist, role-specific permissions are valid, and evaluation fields are well formed. A scenario that raises an exception or returns a validation error is repaired before being passed to the LLM review panels. B.4.2 LLM S TRUCTURAL VALIDATION

You are an evaluation judge for HAS-Bench (Human-Agent System Benchmark). Your task is to evaluate the justification of an Agent-to-User control request, i.e., a moment when the Agent pauses execution and asks the User to approve, deny, or otherwise authorize a proposed action. It should proceed autonomously on routine or low-risk actions. Over-requesting, such as asking permission for trivial reads or internal reasoning, introduces unnecessary friction. C.8 LLM-J UDGE PROTOCOL AND SAFEGUARDS 

LLM judges are used only where the evaluation target is a natural-language outcome or an interaction-quality judgment that cannot be computed directly from the environment state. De-terministic environment assertions, delivery checks, and the Safety Rate for protected actions are computed from task state, execution traces, and domain registries rather than from an LLM judge. For judged quantities, we use fixed rubrics and deterministic judge settings. The judge prompt is provided with the task specification, the relevant execution trace or output, and the metric-specific rubric. For process metrics, the rubrics in Appendix B.5 define the event type and score scale: CQS scores clarification quality, FUR scores whether feedback is incorporated, CRJ scores the justifica-tion for control requests, and ASR scores whether agent-side actions respect safety and authorization constraints. The same judge backbone, GPT-4.1, is used throughout the main experiments unless otherwise specified. We treat judge outputs as auditable measurements rather than opaque ground truth. The prompts are evidence-preserving: the judge is asked to evaluate only the provided task, trace, and output, and not to infer unstated facts. The scenario-level human verification in Table 13 checks the underlying benchmark artifacts, while this judge protocol constrains how natural-language and trace-level met-rics are produced.
